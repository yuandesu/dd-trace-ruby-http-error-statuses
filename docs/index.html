DD_TRACE_HTTP_SERVER_ERROR_STATUSES

dd-trace-ruby で
4xx エラーを追跡する

APM Traces Explorer と Error Tracking に 4xx を表示するまで

Ruby 3.3 Rack dd-trace-rb 2.x Datadog Agent 7 Docker Compose
はじめに

Datadog のデフォルト動作を理解する

デフォルトでエラーになるもの

Rack::TraceMiddleware はレスポンスを自動で計装しますが、 デフォルトでは 5xx（500〜599）のみを エラーとしてマークします。

500 ✗ エラー扱い 503 ✗ エラー扱い

デフォルトでエラーにならないもの

4xx はデフォルトで通常レスポンスとして扱われます。 APM Traces Explorer に赤く表示されず、 Error Tracking にも Issue は作成されません。

403 → エラーなし 422 → エラーなし
目標

このデモで達成すること

1

APM Traces Explorer でエラーとして表示する
span.status = 1 がセットされると、スパンが赤くハイライトされ、エラーとして検索・フィルタできるようになります。

2

Error Tracking に Issue として集約する
error.type / error.message / error.stack の 3 つが セットされていると、同種エラーが自動でグルーピングされ Issue として管理できます。

3

3 つのアプローチを比較して違いを理解する
「デフォルト」「環境変数で設定」「コードで設定」の 3 パターンで どこまで追跡できるかを実際に確認します。

Service ① — before

何も設定しない場合（デフォルト）

docker-compose.yml
app-before:
  environment:
    DD_SERVICE: http-status-before
    DD_ENV: local
    DD_AGENT_HOST: datadog-agent

# DD_TRACE_HTTP_SERVER_ERROR_STATUSES
# → 未設定 = デフォルト (500-599)

環境変数を一切追加しない最もシンプルな構成です。 Rack の自動計装のみが動作します。

403 / 422 を返しても span.status は 0 のまま。 エラーとして記録されません。

結果
span.status 0（エラーなし）
error.type —
APM Traces Explorer ✗ エラーなし
Error Tracking ✗ Issue なし

いつ使うか： 4xx はすべて「正常な業務レスポンス」として扱いたい場合。 例：認証エラーや入力バリデーションエラーをノイズとして除外したいとき。

Service ② — after

DD_TRACE_HTTP_SERVER_ERROR_STATUSES を設定する

docker-compose.yml
app-after:
  environment:
    DD_SERVICE: http-status-after
    DD_ENV: local
    DD_AGENT_HOST: datadog-agent
    # 403, 422 をエラー範囲に追加
    DD_TRACE_HTTP_SERVER_ERROR_STATUSES:
      "403,422,500-599"

この環境変数を設定するだけで、 コード変更なしで 4xx をエラーとしてマークできます。

ただし Rack Middleware が行うのは span.status=1 の設定のみ。 error.type などのタグは付与されません。

結果
span.status 1（エラー）
error.type / message / stack — 付与されない
APM Traces Explorer ✓ エラー表示
Error Tracking ✗ Issue なし

いつ使うか： APM での可視化（エラーレート監視・アラート）だけが目的の場合。 Error Tracking は不要で、特定の 4xx を APM 上でエラーカウントしたいとき。

Service ③ — for-error-tracking

span.set_error をコードで呼び出す

config.ru（抜粋）
when '/forbidden'
  span = Datadog::Tracing.active_span
  if span
    err = StandardError.new('Access forbidden')
    err.set_backtrace(caller)
    # status=1 + error タグを一括セット
    span.set_error(err)
  end
  [403, headers, body]

span.set_error(exception) は内部で span.status=1 と error.type / error.message / error.stack を 一括でセットします。

環境変数の追加は不要。アプリコードで制御するため、 特定の条件だけエラーにしたい場合に最適です。

結果
span.status 1（エラー）
error.type "StandardError"
error.message "Access forbidden"
APM Traces Explorer ✓ エラー表示
Error Tracking ✓ Issue 作成

3 サービスの比較

ポイント： Error Tracking に Issue を作成するには error.type / error.message / error.stack の 3 つが必須です。 DD_TRACE_HTTP_SERVER_ERROR_STATUSES だけでは span.status=1 のみが設定されるため、Error Tracking には届きません。
