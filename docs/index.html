DD_TRACE_HTTP_SERVER_ERROR_STATUSES と Error Tracking

4xx レスポンスを APM Traces Explorer と Error Tracking で追跡するための 3 パターン比較デモ

💡 このデモについて

デフォルト動作

Datadog Ruby トレーサーは、デフォルトで 5xx レスポンスのみ を エラースパンとしてマークします。
4xx（403、422 など）は通常レスポンスとして扱われ、APM にもエラーとして表示されません。

DD_TRACE_HTTP_SERVER_ERROR_STATUSES

この環境変数を設定することで、特定の 4xx ステータスコードをエラーとしてマーク できます。 ただし span.status=1 のみが設定され、 error.type / error.message / error.stack は付与されません。

span.set_error

コード内で span.set_error(exception) を呼び出すことで、 span.status=1 に加えて error タグすべてを一括でセット できます。 Error Tracking に Issue を作成するにはこの方法が必要です。

🔀 403 / 422 レスポンスのトレースフロー

DD_TRACE_HTTP_SERVER_ERROR_STATUSES が未設定のため、 デフォルトのエラー範囲（500-599）のみが適用されます。

403 / 422 はエラー範囲外のため、 Rack::TraceMiddleware は span.status を 0 のままにします。 エラータグも付与されないため、APM・Error Tracking ともに何も記録されません。

DD_TRACE_HTTP_SERVER_ERROR_STATUSES=403,422,500-599 を設定することで、 403 / 422 がエラー範囲に含まれます。

Rack::TraceMiddleware はレスポンス後にステータスコードを確認し、 span.status=1 をセットします。 ただし error.type / error.message / error.stack は付与されないため、Error Tracking には届きません。

コード内で span.set_error(exception) を明示的に呼び出します。

span.set_error は span.status=1 と同時に error.type / error.message / error.stack を一括でセットします。 これにより APM Traces Explorer のエラー表示と Error Tracking の Issue 作成が両方有効になります。

⚙️ 各サービスの設定

before — docker-compose.yml
# DD_TRACE_HTTP_SERVER_ERROR_STATUSES 未設定
app-before:
  environment:
    DD_SERVICE: http-status-before
    DD_ENV: local
    DD_AGENT_HOST: datadog-agent
after — docker-compose.yml
app-after:
  environment:
    DD_SERVICE: http-status-after
    DD_ENV: local
    DD_AGENT_HOST: datadog-agent
    # 403,422 もエラー範囲に追加
    DD_TRACE_HTTP_SERVER_ERROR_STATUSES:
      "403,422,500-599"
for-error-tracking — config.ru
# span.set_error でエラータグを付与
when '/forbidden'
  span = Datadog::Tracing.active_span
  if span
    err = StandardError.new('Access forbidden')
    err.set_backtrace(caller)
    # status=1 + error タグを一括セット
    span.set_error(err)
  end
  [403, ...]
